# Quickstart: Validating Product Catalog Browsing

How to prove this feature works end-to-end once implemented. This is a validation guide, not an implementation guide — see `tasks.md` (generated by `/speckit-tasks`) for build steps.

## Prerequisites

- Node.js (LTS) and a package manager installed.
- Repository dependencies installed (`npm install` once `package.json` exists per `plan.md`'s Project Structure).
- Product content entries present under `src/content/products/` per `contracts/product-content-schema.md`, reconciled against `cliente/productos.md` (all ~24 products, all 6 categories represented).

## Build and preview

```bash
npm run build
npm run preview
```

Expected: build completes with no Content Collection schema errors (a missing `presentations` entry, invalid `category`, or non-positive `priceMXN` fails the build per the content contract — this is the intended safety net for FR-003/FR-005/FR-007).

## Scenario walkthroughs (map to spec User Stories)

### US1 — Browse the full catalog by category

1. Open the catalog entry point (`/catalogo`).
2. Confirm all 6 categories render with a visible heading: Cheesecakes, Gelatinas, Panqués, Pasteles, Postres, Sobre Pedido.
3. Use a category link from `CategoryNav` and confirm the page jumps directly to that category's section.
4. Spot-check one product per category for name, photo, and description (FR-004).

**Automated coverage**: `tests/e2e/catalog-browsing.spec.ts`.

### US2 — Compare presentations and prices

1. Open a multi-presentation product (e.g., "Oreo": Grande/Mediano/Individual).
2. Confirm all three size/portion labels and their distinct prices (`$540`, `$340`, `$75`) are visible without any click/expand interaction.
3. Open a single-presentation product (e.g., "Rosca de Naranja").
4. Confirm it renders its one size/price plainly, with no empty comparison layout.
5. Trigger the product's WhatsApp CTA and confirm the pre-filled message names the product (and presentation, if selected) per `contracts/whatsapp-cta-contract.md`.

**Automated coverage**: `tests/e2e/presentation-pricing.spec.ts`.

### US3 — Recognize made-to-order products

1. Navigate to the Sobre Pedido category.
2. Confirm the category itself is labeled as made-to-order.
3. Open any product within it (e.g., "Panqué de Plátano") and confirm the advance-notice text ("3 días") is visible on the product.
4. Trigger its WhatsApp CTA and confirm the message reflects the made-to-order expectation.

**Automated coverage**: `tests/e2e/sobre-pedido.spec.ts`.

## Resilience checks

- **No-JS check**: Re-run the US1–US3 walkthroughs with JavaScript disabled (Playwright: `context.newPage({ javaScriptEnabled: false })` equivalent, or a browser dev-tools JS toggle for manual spot-checks). Every product name, description, presentation, price, and Sobre Pedido notice must remain visible (FR-013).
- **Accessibility check**: Run `@axe-core/playwright` against the catalog page; zero critical/serious violations (constitution WCAG AA target).
- **Performance check**: Run Lighthouse (mobile + desktop) against the built catalog page; Performance/Accessibility/Best Practices/SEO each ≥ 90 (SC-007, constitution Performance principle).

## Done when

- All three `tests/e2e/*.spec.ts` suites pass, including their no-JS variants.
- Lighthouse and axe checks meet the thresholds above.
- Every product in `cliente/productos.md` has a corresponding, correctly categorized content entry with all its real presentations and prices.
